The product bet

I didn't build everything shallowly.
I built one opinionated thing deeply.

The brief rewards sharp scoping. So instead of a generic CRM with twenty half-features, I committed to a single point of view: a true AI agent that takes a broad marketing goal and executes the campaign end-to-end — gated by human approval at the one moment that matters.

Goal in, plan out

The marketer types intent. The agent returns a named campaign, a segment with reasoning, a recommended channel, and two message variants — not a blank form.

Approval is the guardrail

The agent can draft but cannot send. A campaign must pass draft → approved by a human before a single message leaves. AI proposes, human disposes.

Real async lifecycle

Sends flow through a real Redis/BullMQ queue to a separate stubbed channel service that calls back with delivered / opened / clicked / converted receipts.

The channel service · the part that matters most

Modeling the full communication lifecycle

A separate stubbed service simulates a real provider. It accepts a send, then asynchronously calls back with each lifecycle event. Some messages fail. Some convert. Receipts can arrive out of order — and the CRM stays correct.

Idempotent ingestion

Every receipt carries a unique event_id. Duplicates are detected and dropped — retries and at-least-once delivery never double-count.

Out-of-order safe

A status rank map means state only ever advances. A late "delivered" can't overwrite a "clicked". Highest lifecycle state wins.

Retries + reconciliation

Jobs retry with exponential backoff; failed is terminal. When all messages reach a terminal state, the campaign auto-completes.

Code walkthrough · the critical 20 lines

How a receipt safely updates state

  • Dedupe by event_id
    First line of defense — a unique index makes replays a no-op.
  • Validate the triple
    Communication, campaign and customer must all match, or the receipt is rejected.
  • Monotonic state via rank
    Only advance status if the new rank ≥ current — except failed, which always wins.
  • Attribute revenue on convert
    Conversions add order_value to attributed revenue, then reconcile the campaign.

AI woven in, not bolted on

Intelligence at three layers

1 · Agentic planning

Claude turns a free-text goal into a structured plan: campaign name, segment rules + reasoning, channel, and message variants. Every output is sanitized against an allow-list before it touches the DB.

2 · Retrieval (RAG)

Each shopper becomes a document; embeddings rank who best matches the goal by cosine similarity — grounding the plan in real audience data, not vibes.

3 · Per-shopper summaries

Each customer profile gets an AI "best next step" line, so the marketer sees intelligence everywhere, not just in the agent.

Transparent tool-trace

Every run persists its tool calls to agent_runs — get_customer_summary → retrieve_insights → preview_segment → draft_variants. The agent's reasoning is auditable.

Graceful degradation

If the model returns bad JSON or a key is missing, a deterministic planner takes over and the model status is recorded — the product never hard-fails.

Model selector

A settings UI lets you pick across Claude, GPT and Gemini families — the architecture treats the model as a swappable component.

System design · explicit tradeoffs

"I'd do X at scale, I did Y for this scope"

What I built for this scope
  • Synchronous fan-out in the send loop — simple, readable, fine for demo-sized audiences.
  • Channel callbacks fired via in-process timers to simulate async provider latency.
  • In-app segment evaluation in Python; SQL audience query in the worker.
  • Embeddings recomputed per run — no vector store, negligible cost at this size.
What I'd change at real volume
  • Push the per-recipient fan-out fully into BullMQ jobs with concurrency + rate limits per channel.
  • Move callbacks to a dedicated channel queue with a dead-letter queue for poison receipts.
  • Materialize segments & precompute funnel counts to avoid recomputing on every read.
  • Persist embeddings in pgvector; add a webhook signature + idempotency key TTL.

  • Review every diff
    AI output was read, questioned and refactored — never merged blind. I can defend every line.

    Docker Compose spins up the whole stack locally — change, run, observe, iterate in minutes.
